# 软件设计基础与结构化设计

这份文档覆盖概要设计、详细设计、内聚、耦合、结构化设计、N-S 图、PAD 图、体系结构、设计模式和程序风格。

## 一、软件设计分两层

软件设计通常分为概要设计和详细设计。

| 阶段 | 核心问题 | 主要任务 | 常见输出 |
|---|---|---|---|
| 概要设计/总体设计 | 系统整体如何组织 | 确定总体结构、模块划分、模块间关系、接口 | 优化的软件构造图、概要设计说明书 |
| 详细设计 | 每个模块内部如何实现 | 确定模块算法和数据结构 | 详细设计说明书、流程图、N-S 图、PAD 图、伪码 |

高频选择：

1. 软件总体设计结束后得到优化的软件构造图。
2. 软件概要设计阶段的主要任务是确定软件系统的总体结构和模块间关系。
3. 软件详细设计阶段的主要任务是确定模块的算法和数据结构。
4. 集成测试主要依据概要设计说明书。

## 二、模块独立性

模块独立性是模块化设计的核心目标。它由两个标准衡量：

1. 内聚度：模块内部各成分之间联系的紧密程度。
2. 耦合度：模块之间相互依赖的紧密程度。

设计目标：高内聚、低耦合。

内聚和耦合大多属于定性标准，不是一个精确数值。

## 三、内聚类型

内聚由低到高：

```text
偶然/巧合内聚 < 逻辑内聚 < 时间内聚 < 过程内聚 < 通信内聚 < 顺序内聚 < 功能内聚
```

| 内聚类型 | 特征 | 题眼 |
|---|---|---|
| 偶然/巧合内聚 | 模块内各处理元素没有联系，只是凑在一起 | “没什么联系” |
| 逻辑内聚 | 逻辑上相似的一组功能，由参数决定执行哪一个 | “由控制参数选择功能” |
| 时间内聚 | 需要在同一时间执行的动作放在一起 | “初始化”“收尾” |
| 过程内聚 | 多个处理必须按特定次序执行，但数据联系不强 | “按特定次序执行” |
| 通信内聚 | 所有处理操作同一数据结构或同一输入/输出 | “对数据区 X 写和读” |
| 顺序内聚 | 前一处理的输出作为后一处理的输入 | “输出作为输入” |
| 功能内聚 | 全部成分共同完成一个单一功能 | “一个且仅一个功能” |

真题例子：

1. 模块内部各处理必须按特定次序执行，是过程内聚。
2. 两个处理分别对同一数据区 X 写数据和读数据，是通信内聚。
3. 给出的选项中没有功能内聚时，顺序内聚通常是最高内聚。
4. 最差的内聚类型是偶然内聚/巧合内聚。

## 四、耦合类型

耦合由低到高常记为：

```text
数据耦合 < 标记耦合 < 控制耦合 < 外部耦合 < 公共耦合 < 内容耦合
```

| 耦合类型 | 特征 | 题眼 |
|---|---|---|
| 数据耦合 | 模块间只通过参数传递基本类型数据 | “基本参数”“简单数据项” |
| 标记耦合/特征耦合 | 传递完整数据结构，但只使用其中一部分 | “传记录/对象，只用部分字段” |
| 控制耦合 | 传递控制标志，影响被调模块内部逻辑 | “flag/mode/控制信号” |
| 外部耦合 | 多个模块依赖外部格式、设备、协议等 | “外部接口/设备格式” |
| 公共耦合 | 多个模块共享全局数据或公共数据区 | “全局变量”“公共数据区” |
| 内容耦合 | 一个模块直接访问或修改另一个模块内部数据/代码 | “直接修改另一模块数据” |

真题例子：

1. 通过参数传递基本类型数据，是数据耦合。
2. 直接修改另一个模块的数据，是内容耦合，不是数据耦合。
3. 数据耦合不是最差耦合，内容耦合才是最差。

答题模板：

> 模块耦合是不同模块之间相互依赖的程度。耦合越低，模块独立性越强，系统越容易理解、修改、测试和复用。

## 五、高内聚、低耦合的含义

高内聚：一个模块内部元素围绕同一个明确目标组织，职责单一，功能集中。

低耦合：模块之间依赖尽可能少，通过清晰接口交换必要数据，不直接访问彼此内部细节。

优点：

1. 修改局部模块不容易影响其他模块。
2. 便于独立开发、测试和复用。
3. 系统更容易理解和维护。
4. 降低缺陷传播和变更风险。

实现建议：

1. 优先使用数据耦合。
2. 少用控制标志。
3. 避免全局变量和共享公共数据。
4. 禁止直接访问别的模块内部数据。
5. 模块职责尽量单一。

## 六、结构化方法

结构化方法包括结构化分析、结构化设计和结构化程序设计。

核心指导原则：自顶向下、逐步求精。

结构化程序设计只使用三种基本控制结构：

1. 顺序结构：语句按先后顺序执行。
2. 选择结构：根据条件选择分支，如 if-else。
3. 循环结构：重复执行，如 while、for。

每个基本结构只有一个入口和一个出口。

## 七、N-S 图与 PAD 图

N-S 图又称盒图，用嵌套盒子表示顺序、选择和循环结构。它不使用任意跳转箭头，因此能避免 GOTO 转移，符合结构化程序设计思想。

PAD 图又称问题分析图，用树形结构表示程序逻辑，同样强调顺序、选择、循环，避免非结构化跳转。

真题判断：描述模块内部处理流程时，N-S 图和 PAD 图相比传统程序流程图的优点是避免 GOTO 转移，遵循结构化程序设计思想。

伪码转 N-S/PAD 的基本步骤：

1. BEGIN/END 对应整体外框。
2. 连续语句 S1、S2 顺序排列。
3. IF a > 5 THEN S3 ELSE S4 对应选择结构。
4. WHILE n <= 5 DO 对应循环结构。
5. 循环体内部顺序放 S5、n = n + 1。

## 八、DFD 到模块结构图

结构化设计可根据 DFD 映射模块结构图 MSD。

变换分析思想：

1. 把 DFD 分为输入流、变换中心、输出流。
2. 模块结构的主模块通常由 DFD 的变换中心映射而来。
3. 输入流映射为输入模块，输出流映射为输出模块。
4. 主模块协调输入模块、变换处理模块和输出模块。

真题判断：使用 MSD 的变换分析方法建立模块结构时，主模块由 DFD 的变换中心部分映射而来，这句话是对的。

## 九、软件体系结构

### 三层结构

三层结构包括：

1. 表示层：负责用户界面、输入输出和展示。
2. 业务逻辑层：负责业务规则、流程处理和系统核心逻辑。
3. 数据层：负责数据存储、查询、更新和持久化。

C/S 和 P2P 模式中都可能采用三层结构。

### MVC

MVC 是一种典型体系结构：

1. Model 模型：数据和业务逻辑，对应数据层。
2. View 视图：界面展示，对应表示层。
3. Controller 控制器：接收输入、协调模型和视图，对应控制逻辑层。

真题判断：MVC 包括表示层、控制逻辑层和数据层，这句话按真题答案是对的。

### 事件驱动风格

事件驱动风格属于控制模型，通过外部事件驱动系统执行。

典型两类：

1. 发布/订阅模型：发布者把事件发布到事件通道，订阅者订阅感兴趣事件，事件分发机制自动通知订阅者。发布者和订阅者解耦。
2. 显式事件处理模型/回调模型：组件注册事件处理器，事件发生时调度器调用对应处理函数。

## 十、设计模式

设计模式是在软件开发中针对反复出现的问题总结出的通用解决方案，本质是复用好的设计，而不是直接复用代码。

真题常见两种：

1. 单例模式：保证一个类只有一个实例，并提供全局访问点。适合日志记录器、配置管理器、数据库连接池等需要唯一实例的资源。
2. 工厂模式：把对象创建逻辑封装到工厂中，客户端不直接 new 具体类。适合根据条件创建不同对象，降低对象创建和使用之间的耦合。

真题陷阱：

1. 设计模式不是为了让程序运行速度最快。
2. 设计模式不是只适用于某一种编程语言。
3. 设计模式的重点是复用成熟设计，提高可维护性、可扩展性和复用性。

## 十一、E-R 模型

E-R 模型是数据库概念设计模型，基本成分：

1. 实体：现实世界中可独立存在的事物。
2. 属性：实体的特征。
3. 联系：实体之间的关联。

真题答案：E-R 模型包括实体、联系、属性。

## 十二、编程风格

编程风格包括命名、格式、注释、结构、代码组织等。

良好的编程风格可以提高可读性、可维护性，也可能间接影响运行效率。例如清晰结构减少冗余逻辑，规范控制结构降低错误概率。

输入结束判断不要绝对化。计数方法适合输入数量已知的场景；文件结束符、输入序列结束符适合输入数量不确定的场景。
