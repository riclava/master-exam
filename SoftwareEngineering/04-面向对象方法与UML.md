# 面向对象方法与 UML

这份文档覆盖类与对象、封装、继承、多态、消息传递、UML 关系、用例图、顺序图、状态图、4+1 视图，以及面向对象分析三大模型。

## 一、类和对象

类是对一类具有相同属性和操作的对象的抽象描述，是模板。

对象是类的实例，是具体存在的实体。

必背关系：

```text
类是对象的抽象；对象是类的实例。
类与对象是抽象与具体的关系。
```

真题陷阱：

1. “类是对象的实例”是错的，应该是“对象是类的实例”。
2. 不同对象应具有不同标识。对象标识在生命周期中保持唯一。
3. 程序不是简单的“对象集合”，面向对象程序强调对象及其交互。
4. 程序运行时不是所有对象都处于活动状态；对象可能未创建、等待、静默或已销毁。

## 二、面向对象三大特性

| 特性 | 含义 | 高频题眼 |
|---|---|---|
| 封装 | 把数据和操作结合为整体，隐藏内部实现，只暴露公共接口 | 信息隐蔽通过封装实现 |
| 继承 | 子类复用父类属性和方法，形成一般/特殊层次 | 子类共享父类属性和方法 |
| 多态 | 不同类对象对同一消息作出不同响应 | 同一方法调用表现不同 |

对象把数据和处理数据的操作结合为一体，这体现封装。

信息隐蔽通过封装实现。只有公共界面成员才允许外部使用，这是信息隐蔽原则。

最小界面原则：当且仅当一个操作对类实例用户有用时，才应作为公共界面成员。

## 三、消息传递

面向对象方法中，对象之间的交互通过消息传递实现。

一个消息通常包括：

1. 接收消息的对象。
2. 接收对象要执行的方法/操作。
3. 方法所需参数。

消息传递不等于传统过程调用：

| 对比项 | 消息传递 | 过程调用 |
|---|---|---|
| 主体 | 对象向对象发消息 | 调用者调用函数/过程 |
| 依赖 | 依赖对象公共接口，不关心内部实现 | 依赖具体函数名、参数、返回值 |
| 动态性 | 同一消息可被不同对象不同响应，支持多态 | 通常静态绑定，行为固定 |
| 目标 | 对象协作 | 功能复用 |

真题陷阱：接收消息的对象不一定必须回答发送者。通知型消息可能只触发动作，不返回结果。

## 四、重载、重写、多态

重载 Overload：同一个类中多个方法名相同，但参数个数、类型或顺序不同，方法体可不同。

重写 Override：子类重新实现父类已有方法。

多态 Polymorphism：不同对象对同一消息产生不同响应。重载常被称为编译时多态，重写结合动态绑定常体现运行时多态。

真题中“采用一样的方法名，重新写方法体”按给定答案选重载；复习时仍要知道严格 OO 语境下“子类重新写父类方法体”通常叫重写。

## 五、UML 类关系

| 关系 | 含义 | 题眼 |
|---|---|---|
| 关联 Association | 类/对象之间长期、稳定的语义联系 | 学生选课、教师授课 |
| 聚合 Aggregation | 弱整体-部分，部分可独立存在 | 学校-教师、剧组-演员 |
| 组合 Composition | 强整体-部分，部分生命周期依赖整体 | 订单-订单项、公司-部门 |
| 泛化 Generalization | 一般/特殊，即继承关系 | 父类-子类，子类共享父类属性方法 |
| 依赖 Dependency | 一个类临时使用另一个类 | 方法参数、局部变量、调用服务 |
| 实现 Realization | 类实现接口 | 支付接口-微信支付类 |

高频辨析：

1. “整体/部分”不是依赖关系，通常是聚合或组合。
2. 部分可以脱离整体独立存在，是聚合。
3. 部分不能脱离整体存在，生命周期依赖整体，是组合。
4. 泛化描述一般化/特殊化关系，本质是继承。
5. 关联是建立类之间关系的一种手段。

例子：

学校和教师：聚合。教师可离开某学校，仍独立存在。

剧组和演员：聚合。演员可属于多个剧组。

对象 a 是对象 b 的组成部分：b 是整体，a 是部分。把 a 说成整体是错的。

## 六、面向对象分析与结构化分析的区别

两者都使用抽象，都目标明确，都需要考虑数据结构。区别在于分析视角不同。

| 方法 | 基点 | 建模重点 |
|---|---|---|
| 结构化分析 SA | 功能、过程、数据流 | 输入、加工、输出、数据存储 |
| 面向对象分析 OOA | 问题域中的实体/对象 | 类、对象、属性、操作、关系和交互 |

真题答案：面向对象分析以问题域的成分为基点构造模块，而结构化分析不是。

## 七、面向对象分析三大模型

| 模型 | 描述 | 常用图 |
|---|---|---|
| 对象模型 | 静态结构，系统中有什么类/对象及其关系 | 类图、对象图 |
| 动态模型 | 对象随事件发生的状态变化和交互时序 | 状态图、顺序图 |
| 功能模型 | 系统应完成的功能和数据流向 | 用例图、数据流图 |

三者关系：

1. 对象模型是基础，提供类、属性和关系。
2. 动态模型描述对象如何响应事件、发送消息、发生状态变化。
3. 功能模型说明系统要做什么，引导对象模型和动态模型的范围。
4. 三者从静态、动态、功能三个角度共同构成完整分析模型。

## 八、UML 用例图

用例图用于描述系统功能需求，表现参与者与用例之间的关系。

用例图只能宏观展示功能范围，不能详细说明用例内部步骤。通常需要辅以用例规格说明或场景。

场景是从用户角度描述某个功能处理的一串事件序列。

真题判断：描述系统功能时常使用用例图建模，但需要辅以规格说明，即用例实现的场景，这句话是对的。

## 九、UML 顺序图

顺序图是一种交互图，用于描述一组对象之间按时间顺序发送的消息。

顺序图常见元素：

1. 参与者：用户、管理员、教师等外部角色。
2. 边界对象：界面、浏览器、终端。
3. 控制对象：登录控制器、销售控制器等。
4. 实体对象/数据存储：用户信息文件、商品数据库、订单文件。
5. 消息：请求、查询、返回、更新、确认等。

答题套路：

1. 参与者发起请求。
2. 界面或系统接收请求。
3. 控制对象查询数据文件。
4. 根据查询结果返回成功或失败。
5. 如需更新，则写入数据文件。
6. 系统向参与者反馈结果。

历年常见顺序图交互：

1. Web 用户登录系统。
2. 图书销售中录入销售信息。
3. 图书管理中借阅图书。
4. 在线购物中添加商品到购物车。

## 十、UML 状态图

状态图用于描述单个对象或系统在生命周期中受事件驱动发生的状态变化。

作用：

1. 刻画对象动态行为。
2. 发现遗漏状态或非法转换。
3. 支持状态覆盖、转移覆盖等测试设计。
4. 指导代码实现，尤其是状态模式和业务流程实现。

真题常考补状态名：

| 业务 | 常见状态 |
|---|---|
| 图书借阅 | 显示首页、借阅图书、校验图书 ID、检查是否可借、查询读者信息、显示借阅成功、是否继续借阅 |
| 成绩查询 | 要求学号、校验学号、要求课程名、校验课程名、查询选课文件、显示成绩、询问用户操作 |
| 地铁购票 | 购票、验证起点站、验证终点站、确认购票信息、验证支付成功、显示购票成功 |

状态图题的关键：

1. 状态通常是系统正在做的事，如“校验学号”“查询文件”。
2. 转移上的文字通常是事件或条件，如“学号正确”“查询失败”。
3. do: 后面描述进入该状态后持续执行或立即执行的动作。

## 十一、4+1 视图模型

面向对象 4+1 视图模型包括：

1. 逻辑视图：类、对象和关系，主要支持功能需求。
2. 物理视图：硬件部署、节点和网络。
3. 开发视图：代码结构、模块、包、构件。
4. 进程视图：进程、线程、并发、通信等运行时行为。
5. 场景视图/用例视图：用例和场景，用来串联并验证其他四个视图。

2025 回忆版考填空：逻辑视图、物理视图、开发视图、进程视图、场景视图。
