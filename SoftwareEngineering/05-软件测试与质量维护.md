# 软件测试、质量与维护

这份文档覆盖软件测试目的、测试阶段、黑盒/白盒、测试用例设计、覆盖准则、McCabe 环路复杂度、质量审查和软件维护。

## 一、软件测试的目的

按 G.J. Myers 的观点，软件测试的目的不是证明软件没有错误，而是以尽可能少的代价、尽可能短的时间，发现尽可能多的错误。

必须背：

1. 测试是为了发现错误。
2. 测试可以验证软件是否满足需求。
3. 测试结果可用于评估软件质量和风险。
4. 测试不能证明软件完全无错。

真题判断：测试没有发现问题，不能证明软件没有错误。

## 二、测试原则

常考答题点：

1. 尽早测试：从需求阶段开始规划测试，越早发现缺陷，修复成本越低。
2. 全面测试：覆盖正常、异常、边界、性能、可靠性等场景。
3. 重点测试：资源有限时重点测试关键功能、复杂模块和高风险区域。
4. 独立测试：测试最好由相对独立人员承担，减少主观偏差。
5. 及时反馈：缺陷要记录、跟踪、反馈和验证修复。
6. 回归测试：修改后确认原有功能未被破坏。
7. 制定测试计划：明确目标、范围、方法、资源、进度。
8. 保留测试记录：便于追溯、复盘和质量评估。

## 三、测试阶段

| 阶段 | 关注点 | 主要依据 | 高频误区 |
|---|---|---|---|
| 单元测试 | 单个模块内部逻辑 | 详细设计说明书、代码 | 不是主要发现模块接口之间错误 |
| 集成测试 | 模块接口、模块协作、组合后的结构 | 概要设计说明书 | 不是主要依据需求规格说明书 |
| 系统测试 | 整个系统是否满足规格 | 需求规格说明书 | 不只测单个模块 |
| 验收测试 | 用户是否接受，是否满足实际需求 | 用户需求、合同、验收标准 | 更接近确认 Validation |

高频判断：单元测试的主要目的不是发现模块接口之间的错误；接口错误主要在集成测试发现。

## 四、黑盒测试与白盒测试

| 项目 | 黑盒测试 | 白盒测试 |
|---|---|---|
| 测试视角 | 不关心内部实现，把程序看作黑盒 | 了解内部逻辑、结构和路径 |
| 主要依据 | 功能规格说明、需求说明 | 程序代码、控制流图、逻辑结构 |
| 关注点 | 输入输出、功能是否正确 | 语句、分支、路径、条件是否覆盖 |
| 常见方法 | 等价类划分、边界值分析、因果图、判定表 | 语句覆盖、判定覆盖、条件覆盖、路径覆盖、基本路径覆盖 |

真题高频：

1. 黑盒测试主要考虑系统功能。
2. 黑盒测试主要根据功能规格说明设计用例。
3. 边界值分析法是黑盒测试法，不是白盒测试法。
4. 等价类划分和边界值分析通常没有系统考虑输入条件之间的组合关系。

若要考虑多个输入条件组合，可用判定表、因果图、正交实验法。

## 五、测试用例

测试用例不能只选合理输入，还必须包括：

1. 合理输入：验证正常功能。
2. 不合理输入：验证错误处理和容错能力。
3. 边界值：验证边界附近是否出错。
4. 特殊值：空值、零值、重复值、超长值、非法格式等。

高频判断：“测试用例只需选用合理输入数据”是错的。

## 六、覆盖准则

### 语句覆盖

要求程序中每条可执行语句至少执行一次。

例：求三个整数 a、b、c 最小值。

```text
min = a
if b < min then min = b
if c < min then min = c
输出 min
```

为覆盖两个赋值语句，可用：

```text
a = 3, b = 2, c = 1
预期输出：1
```

一个用例即可覆盖所有语句。

### 判定覆盖/分支覆盖

要求每个判定的真假分支至少各执行一次。

### 条件覆盖

要求每个简单条件的真、假取值至少出现一次。

### 条件组合覆盖

要求每个判定中所有简单条件的真假组合都至少出现一次。

公式：n 个简单条件至少有 `2^n` 种组合。

例：`(A >= 9) and (B < 0)` 有两个简单条件：

| 组合 | A >= 9 | B < 0 | 整体结果 |
|---|---|---|---|
| 1 | 真 | 真 | 真 |
| 2 | 真 | 假 | 假 |
| 3 | 假 | 真 | 假 |
| 4 | 假 | 假 | 假 |

至少需要 4 个测试用例。

### 基本路径覆盖

基本路径覆盖是一种白盒测试方法。它根据程序控制流图计算环路复杂度，导出线性无关路径集合，并设计测试用例使每条基本路径至少执行一次。

基本路径覆盖通常满足较强的路径覆盖要求，至少会覆盖语句和分支。

## 七、McCabe 环路复杂度

常用计算方法：

```text
V(G) = 判定节点数 + 1
V(G) = E - N + 2
V(G) = 区域数
```

其中 E 是边数，N 是节点数。

真题常直接用“判定节点数 + 1”。

例子：

1. 图书借阅流程有 4 个判定节点，环路复杂度 = 4 + 1 = 5。
2. 成绩查询流程有 4 个判定节点，环路复杂度 = 5。
3. 程序流程图若有 4 个判定节点，基本路径覆盖至少需要 5 条独立路径。

基本路径覆盖答题模板：

1. 识别判定节点数量。
2. 计算 `V(G) = 判定节点数 + 1`。
3. 写出 V(G) 条相互独立路径。
4. 为每条路径设计输入数据和预期结果。

## 八、质量审查与评审

正式技术评审的目标是发现软件产品中的错误，包括需求、设计、代码、测试文档等缺陷。

它不是：

1. 评价程序员工作效率。
2. 记录程序员出错并与绩效挂钩。
3. 在评审会议中直接修改所有错误。

验证 Verification 与确认 Validation：

1. 验证：检查是否正确地做了产品，关注产品与规格、设计、过程的一致性。
2. 确认：检查是否做了正确的产品，关注是否满足用户真实需求。

## 九、软件质量

软件正确性：软件达到预期功能和性能需求、准确执行预期任务的程度。不是“能运行”这么简单，也不等于“绝对没有任何错误”。

软件质量保证 SQA 的目的：为软件过程和软件产品提供保障，使其符合规定需求并遵循既定计划。

软件质量三要素按真题：人员素质、过程、技术。

## 十、软件维护

软件维护是在软件交付后，为纠正错误、适应环境变化、改进性能或可维护性而进行的修改。

四类维护：

| 类型 | 含义 | 示例 |
|---|---|---|
| 纠错性维护 | 修复交付后发现的错误 | 修复崩溃、计算错误 |
| 适应性维护 | 适应环境变化 | 操作系统升级、数据库迁移 |
| 完善性维护 | 增强功能或改善性能 | 新增报表、优化查询速度 |
| 预防性维护 | 提高未来可维护性和可靠性 | 重构代码、清理隐患 |

高频判断：“软件维护不包括纠错”是错的。

## 十一、易错判断速记

边界值分析是黑盒测试，不是白盒测试。

测试不能证明软件无错。

测试用例不能只包含合理输入。

单元测试主要不是测模块接口。

集成测试主要依据概要设计说明书。

基本路径覆盖与 McCabe 环路复杂度相关。

质量控制不是软件配置管理活动。

正式技术评审的目标是发现错误，不是考核个人。
