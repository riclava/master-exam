# 软件过程与项目管理

这份文档覆盖历年真题中的软件生命周期、过程模型、RUP/RAD、CMM/CMMI、配置管理、质量保证、项目估算与风险管理。

## 一、软件生命周期

软件生命周期通常包括：

| 阶段 | 主要任务 | 常见输出 |
|---|---|---|
| 可行性研究与计划 | 判断项目是否值得做、能不能做、怎么做 | 可行性研究报告、项目计划 |
| 需求分析 | 明确功能、性能、接口、约束，建立逻辑模型 | 需求规格说明书、DFD、数据字典、用例模型 |
| 概要设计 | 确定总体结构、模块划分、模块关系、接口 | 优化的软件构造图、概要设计说明书 |
| 详细设计 | 确定模块内部算法和数据结构 | 详细设计说明书、流程图、N-S 图、PAD 图 |
| 编码 | 将详细设计转化为程序 | 源代码、可运行程序 |
| 测试 | 发现错误并验证需求满足程度 | 测试计划、测试用例、测试报告 |
| 维护 | 交付后修改和完善软件 | 维护记录、变更版本 |
| 退役 | 停止使用、迁移数据、替换系统 | 退役计划、迁移方案 |

必背辨析：

需求分析不是确定算法；算法和数据结构属于详细设计。

概要设计不是写代码；它确定系统总体结构和模块接口。

测试计划通常在需求分析阶段制定，因为测试目标要对准需求。

集成测试主要依据概要设计说明书，因为它关注模块接口和模块组合。

## 二、可行性研究

承包或启动软件项目前要做可行性研究，目的是避免盲目投入，提前识别项目风险，判断项目是否具备实施价值。

常考四类可行性：

1. 技术可行性：现有技术、团队能力、软硬件环境能否支持项目实现。
2. 经济可行性：成本、收益、投资回报是否合理。
3. 操作可行性：用户是否能接受和使用，系统是否适合现有业务流程。
4. 社会可行性：是否符合法律法规、行业规范和伦理要求。

答题模板：

> 可行性研究是在项目正式开发前，对技术、经济、操作和社会等方面进行分析，判断项目是否值得实施、能否成功实施，并为后续计划、资源安排和风险控制提供依据。

## 三、过程模型

| 模型 | 核心特征 | 适用场景 | 高频陷阱 |
|---|---|---|---|
| 瀑布模型 | 线性顺序、阶段清楚、文档驱动 | 需求明确且稳定 | 大型复杂系统不一定最适合瀑布 |
| 增量模型 | 将需求分组，每个增量可按瀑布式开发并逐步交付 | 需要早期交付核心功能 | 增量模型强调交付策略，不直接保证模块划分好 |
| 演化模型 | 小步迭代，每步经历需求、设计、实现、验证 | 需求会逐步澄清 | 不是一次性完成全部需求 |
| 螺旋模型 | 瀑布 + 演化 + 风险分析，基于原型 | 大型、高风险、需求不确定项目 | 关键字是风险分析和原型 |
| 喷泉模型 | 面向对象，迭代、无间隙 | 面向对象开发 | 阶段之间没有严格边界 |
| 敏捷模型 | 迭代增量，减少不必要文档，分析/设计/编码/测试结合 | 需求变化快、强调反馈 | 不是不要文档，而是不做无价值的过度文档 |
| RAD | 快速应用开发，短周期，构件化 | 需求理解较好、范围受限、需快速交付 | 常考业务建模、数据建模、流程建模 |

增量模型开发活动常见说法：增量分析、增量设计、增量实现。真题中“增量发布”被作为非标准开发活动干扰项。

螺旋模型四个活动可概括为：规划、风险分析、开发、评估。每转一圈得到一个新版本。

喷泉模型体现面向对象开发中的迭代和无间隙特征。

## 四、RUP

RUP 中常用三要素：

| 问题 | RUP 术语 | 含义 |
|---|---|---|
| 谁做 | 角色 | 执行活动的人或团队职责 |
| 做什么 | 活动/制品 | 活动产生制品，如文档、模型、代码 |
| 怎么做 | 工作流 | 活动的组织方式和过程 |

RUP 的 6 个核心工程工作流：

1. 业务建模：理解业务流程、组织目标和业务上下文。
2. 需求：获取、分析、描述功能需求和非功能需求，建立用例模型。
3. 分析与设计：把需求转化为设计模型，确定架构、类、接口和协作。
4. 实现：编码、单元测试、构建可执行构件。
5. 测试：验证软件是否满足需求，发现缺陷。
6. 部署：安装、配置、培训、交付到实际环境。

RUP 还包含支持工作流，如配置与变更管理、项目管理、环境等。

## 五、RAD 三种建模

RAD 真题要求会答三种建模及活动：

1. 业务建模：定义业务职能之间的信息流，说明哪些数据驱动业务、生成哪些数据、信息去向何处、由谁处理。
2. 数据建模：根据业务建模得到的信息，识别数据对象、属性和对象之间的关系。
3. 流程建模：把数据对象转换为业务功能所需的数据流，描述增删改查等处理。

## 六、软件过程的 5 个基本过程

按不同人员/主体的工作内容分类，软件生存周期中的 5 个基本过程是：

| 基本过程 | 相关方 | 典型活动 |
|---|---|---|
| 获取过程 | 获取方/需求方 | 启动、招标、签订合同、监督供应方、验收 |
| 供应过程 | 供应方 | 投标、签约、计划、执行、交付 |
| 开发过程 | 开发方 | 需求、设计、编码、测试、安装、验收支持 |
| 运行过程 | 运行方 | 运行测试、系统运行、用户支持 |
| 维护过程 | 维护方 | 问题分析、修改实现、维护评审/验收、迁移、退役 |

注意：改善过程属于组织过程，不是基本过程。维护过程在某些分类中也可能作为基本过程出现，按真题答案应背“获取、供应、开发、运行、维护”。

## 七、CMM 与 CMMI

### CMM 五级

| 等级 | 名称 | 核心特征 |
|---:|---|---|
| 1 | 初始级 | 过程无序，依赖个人能力，无稳定规范 |
| 2 | 可重复级 | 建立基本项目管理过程，能重复以往成功经验 |
| 3 | 已定义级 | 过程文档化、标准化，形成组织级标准过程 |
| 4 | 已管理级 | 对过程和产品质量进行定量管理 |
| 5 | 优化级/持续优化级 | 基于量化反馈和新技术持续改进 |

常考关键过程域：

| 等级 | 关键过程域 |
|---|---|
| 1 初始级 | 无 |
| 2 可重复级 | 需求管理、软件项目计划、项目跟踪与监控、软件子合同管理、软件质量保证、软件配置管理 |
| 3 已定义级 | 组织过程焦点、组织过程定义、培训大纲、集成软件管理、软件产品工程、组间协调、同行评审 |
| 4 已管理级 | 定量过程管理、软件质量管理 |
| 5 优化级 | 缺陷预防、技术变更管理、过程变更管理 |

CMM 认为支撑软件质量的三个要素是人员素质、过程、技术。管理常作为干扰项。

### CMMI

CMMI 是能力成熟度模型集成，用于过程改进。它既支持阶段式表示，也支持连续式表示。

连续式能力等级常背 0 到 5：

0. 不完整级
1. 已执行级
2. 已管理级
3. 已定义级
4. 量化管理级
5. 最优化级

真题陷阱：说“CMMI 只支持阶段性改进，不支持连续性改进”是错的。

## 八、质量保证、配置管理和评审

软件质量保证 SQA 是为软件过程和软件产品提供保障，使它们符合规定需求并遵循既定计划。

真题中质量保证四个活动：

1. 过程实现
2. 产品保证
3. 过程保证
4. 质量体系保证

正式技术评审的目标是发现软件产品中的错误，不是评价程序员效率，也不是现场改正错误。

验证与确认：

| 项目 | 验证 Verification | 确认 Validation |
|---|---|---|
| 核心问题 | 是否正确地做了产品 | 是否做了正确的产品 |
| 关注点 | 产品是否符合规格、设计和过程规范 | 产品是否满足用户真实需求 |
| 常用方法 | 评审、走查、单元测试、集成测试 | 系统测试、验收测试、用户试用 |
| 关系 | 验证是确认的基础 | 确认检验最终需求满足度 |

软件配置管理 SCM 是保护伞活动，贯穿整个生命周期，用于控制变更。常见活动包括配置项标识、版本控制、变更控制、配置审计、状态报告。

质量控制不是配置管理活动。质量控制关注测试、评审和缺陷发现；配置管理关注配置项和变更。

## 九、项目估算、风险与进度

最常用的软件规模度量：代码行 LOC 和功能点 FP。

代码行度量简单直观，但依赖语言，通常不适合很早期估算。功能点分析基于外部输入、外部输出、查询、文件等功能规模，适合需求阶段早期估算。

Gantt 图用于项目进度控制。IPO 图用于描述模块输入、处理、输出。功能点分析用于规模估算。挣值分析用于进度和成本绩效监控。

软件开发成本不仅包括直接人力成本，还包括间接人力、设备、外包、办公、电力等经常性消耗。

风险预测主要看两个维度：

1. 风险发生的可能性。
2. 风险发生后的影响程度。

“风险消除方法”属于风险应对，不是风险预测维度。
