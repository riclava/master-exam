# 需求工程与结构化分析

这份文档覆盖需求分析、需求开发/管理、需求层次、结构化分析、DFD、数据字典和建模题的基础能力。

## 一、需求分析到底做什么

需求分析的核心问题是“软件要做什么”，不是“软件怎么实现”。

常考任务：

1. 获取和分析用户需求。
2. 确定功能需求、性能需求、接口需求和约束条件。
3. 建立系统逻辑模型，如 DFD、数据字典、用例模型。
4. 编写需求规格说明书 SRS。
5. 为后续设计、编码、测试提供基准。

常见非需求任务：

| 任务 | 所属阶段 |
|---|---|
| 给出软件处理方案 | 设计阶段 |
| 给出系统模块构造 | 概要设计 |
| 定义模块算法 | 详细设计 |
| 程序编码 | 编码阶段 |
| 确定实现算法 | 详细设计 |

高频判断：需求分析阶段的首要任务是定义用户需求并建立系统逻辑模型。

## 二、需求开发与需求管理

需求工程通常分为需求开发和需求管理。

需求开发强调“从无到有生成需求”：

1. 需求获取：访谈、问卷、观察、原型等方式收集用户需求。
2. 需求分析建模：整理、冲突分析、优先级分析，建立 DFD、用例图等模型。
3. 需求定义：形成规范化需求描述。
4. 需求评审：检查需求正确性、完整性、一致性、可行性。

需求管理强调“需求确定后如何维护”：

1. 需求变更控制。
2. 需求版本管理。
3. 需求状态跟踪。
4. 需求与设计、代码、测试之间的可追踪性。

真题陷阱：软件需求管理不属于需求开发活动。

## 三、需求的层次

软件需求是对待开发软件在功能、性能、属性和约束等方面的陈述，是软件开发的核心依据。

真题中常按两个核心层次答：

1. 功能需求：系统必须提供哪些功能或服务，即“做什么”。
2. 非功能需求：系统要做到什么程度，包括性能、外部接口、质量属性、设计约束等。

非功能需求常见类别：

| 类别 | 示例 |
|---|---|
| 性能需求 | 响应时间、吞吐量、并发数、资源利用率 |
| 外部接口需求 | 用户接口、硬件接口、软件接口、通信接口 |
| 质量属性 | 可靠性、可用性、可维护性、安全性、可移植性 |
| 设计约束 | 编程语言、数据库、平台、标准、法律法规 |

功能和性能必须一起考虑。功能说明系统做什么，性能说明功能执行达到什么效果。

## 四、结构化分析总览

结构化分析 SA 是面向过程的需求分析方法，核心从数据流动角度理解系统功能。

指导原则：自顶向下、逐步求精。

完整结构化分析模型包括：

1. 数据流图 DFD：描述数据来源、流动、加工、存储和去向。
2. 数据字典 DD：定义 DFD 中的数据流、数据存储、数据项和数据结构。
3. 加工说明：描述 DFD 中加工的内部处理规则，可用结构化语言、判定表、判定树。

真题陷阱：结构化分析建立的系统模型不只是数据流图。只说“系统模型就是 DFD”是错的。

## 五、DFD 四种基本元素

| 元素 | 含义 | 图中角色 | 识别方法 |
|---|---|---|---|
| 外部实体 | 系统外部的人、组织或其他系统 | 系统边界外 | 用户、管理员、教师、供应商、银行等 |
| 加工 | 对输入数据进行处理并产生输出 | 系统功能 | 动词短语，如“校验请求”“生成报表” |
| 数据流 | 数据在实体、加工、存储之间移动 | 箭头 | 名词短语，如“订单信息”“登录请求” |
| 数据存储 | 系统内部保存数据的地方 | 文件/表 | 数据库、文件、表，如“用户信息表” |

表达系统边界通常靠外部实体。外部实体位于系统之外，与系统发生数据交换。

DFD 描述数据流动，不描述控制流程。控制流程应由流程图、控制流图或状态图描述。

## 六、DFD 分层

### 顶层 DFD

顶层 DFD 又叫系统环境图或上下文图。它把整个系统看成一个加工，只展示系统与外部实体之间的数据流。

顶层 DFD 主要回答：

1. 系统边界在哪里。
2. 外部实体有哪些。
3. 系统有哪些输入和输出。

顶层 DFD 不展开系统内部处理，不画内部加工细节。真题中“DFD 顶层流图用于指出系统的输入与输出”。

### 0 层 DFD

0 层 DFD 是对顶层系统加工的第一次分解。它把系统拆成若干核心加工，并补充内部数据存储和加工间的数据流。

注意平衡原则：顶层图中外部实体与系统之间的数据流，在 0 层图中仍应能对应，不要无故增加或丢失外部输入输出。

### 1 层 DFD

1 层 DFD 是对 0 层中某个加工的继续分解。比如把“请求校验”分解为“用户信誉校验”和“商品库存校验”。

## 七、数据字典

数据字典用于精确定义 DFD 中出现的数据流和数据存储。

常用写法：

```text
数据流名 = 数据项 + 数据项 + 数据项
重复项 = {数据项 + 数据项}
选择项 = [选项1 | 选项2]
```

例子：

```text
入库图书信息 = 图书名 + 图书编号 + 数量 + 单价 + 入库日期
库存清单 = {图书名 + 图书编号 + 库存量 + 库存金额}
借还书的规定 = [借书信息 | 还书信息]
```

答题时要注意：

1. 数据流名称必须与 DFD 或题干一致。
2. 数据项尽量来自题干原始描述。
3. 不要把外部实体的动作写成数据项，比如“查询”“修改”是动作，不是数据。
4. 顶层 DFD 的数据字典常定义外部输入输出数据流；0 层/1 层可增加数据存储定义。

## 八、加工说明

DFD 只描述“数据怎么流”，不能完整描述加工内部逻辑。加工说明用于补充每个加工如何处理输入数据。

常用工具：

1. 结构化语言：用顺序、选择、循环描述逻辑。
2. 判定表：适合条件组合较多、动作选择明确的场景。
3. 判定树：适合分层条件判断。

可能考点：加工说明、数据字典和 DFD 配合，才构成比较完整的结构化分析模型。

## 九、结构化分析答题步骤

DFD 建模题建议按以下顺序做：

1. 圈外部实体：题干中的人、组织、其他系统通常就是外部实体。
2. 找系统名：题干给出的“某某管理系统”通常就是顶层加工名。
3. 抽数据流：只写名词性数据，如“购买请求”“订单信息”“缺货通知”。
4. 找数据存储：题干里的“文件、表、库、清单、记录”通常是数据存储。
5. 分核心加工：0 层图按核心业务动作分解，如“请求校验”“订单处理”“报表统计”。
6. 检查平衡：外部实体和进出系统的数据流不能和顶层图矛盾。
7. 写数据字典：把题干中的字段拆成数据项。

## 十、历年建模题常见系统

历年题材都很生活化，重点不是业务复杂，而是识别数据流：

| 年份 | 系统 | 主要建模要求 |
|---|---|---|
| 2012 | 图书销售管理系统 | 顶层 DFD、数据字典、顺序图 |
| 2013 | Web 商品购销存系统 | 顶层 DFD、数据存储、登录顺序图 |
| 2014 | 教学管理系统 | 顶层 DFD、数据文件、第一层 DFD |
| 2018 | 在线作业批改系统 | 补顶层 DFD 数据流、作业数据流数据字典 |
| 2022 | 培训中心管理系统 | 顶层 DFD、0 层 DFD |
| 2023 | 图书管理系统 | 顶层 DFD、数据字典、顺序图 |
| 2025 模拟 | 在线购物系统 | 顶层 DFD、数据存储定义、顺序图 |
| 2025 回忆 | 销售管理系统 | 顶层 DFD、0 层 DFD、1 层 DFD |

结论：DFD 是软件工程建模题第一优先级。必须能从文字描述直接画出顶层和 0 层图，并会写数据字典。
